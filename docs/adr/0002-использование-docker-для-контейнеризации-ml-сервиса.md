# 2. Использование Docker для контейнеризации
Дата: 2025-11-30

## Статус
Принято

## Контекст
Сервис должен быть:
- воспроизводимым,
- переносимым,
- подходящим для развёртывания через Terraform,
- совместимым с Yandex Cloud.

Необходимо выбрать технологию упаковки приложения.

## Рассмотренные варианты

### Вариант 1: Docker  *(выбран)*
Плюсы:
- стандартизированная упаковка
- идентичное окружение от CI до продакшена
- простая интеграция с GitHub Actions
- можно использовать Serverless Containers или VM
- healthcheck прямо внутри Dockerfile

Минусы:
- небольшой оверхед

### Вариант 2: Виртуальные машины
Минусы:
- медленно поднимать
- труднее воспроизводить окружение
- тяжеловесно для одного API-сервиса

### Вариант 3: Нативное развертывание
Минусы:
- плохо воспроизводимо
- зависит от хост-окружения

## Решение
Использовать Docker как базовый способ контейнеризации приложения.

## Обоснование
1. **Воспроизводимость окружения** — критично для DevOps.
2. **Совместимость с CI/CD** — GitHub Actions легко собирает и пушит образ.
3. **Интеграция с Terraform** — YCR + Serverless Containers/VM.
4. **Простая локальная разработка** 
5. **Healthcheck в Dockerfile**

## Последствия
- + надёжное развёртывание
- + минимизация ручной настройки серверов
- − знание Docker
